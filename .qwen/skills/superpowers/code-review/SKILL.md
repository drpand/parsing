# Суперспособность: Code Review (Подготовка к ревью)

## Назначение
Самопроверка кода перед отправкой на ревью или коммитом.

## Когда использовать
- Перед коммитом в git
- Перед пушем в репозиторий
- После завершения задачи
- Перед запросом ревью у коллеги

## Алгоритм code review

### Шаг 1: Self-review (самопроверка)
```
1. Прочитать свой код как чужой
2. Задать вопросы:
   - Понятно ли что делает этот код?
   - Можно ли упростить?
   - Нет ли дублирования?
   - Обработаны ли ошибки?
```

### Шаг 2: Чеклист качества
```
[ ] Код следует стилю проекта
[ ] Нет дублирования (DRY)
[ ] Функции маленькие (до 20 строк)
[ ] Названия понятные
[ ] Нет закомментированного кода
[ ] Нет отладочных принтов
[ ] Логи на месте
[ ] Типизация добавлена
```

### Шаг 3: Безопасность
```
[ ] Нет хардкода секретов
[ ] Токены из .env
[ ] Входные данные валидируются
[ ] Ошибки не показывают детали
[ ] SQL инъекции исключены
[ ] Нет уязвимостей
```

### Шаг 4: Тесты
```
[ ] Unit тесты написаны
[ ] Integration тесты есть
[ ] Edge cases покрыты
[ ] Все тесты проходят
[ ] Покрытие не ухудшилось
```

### Шаг 5: Документация
```
[ ] Docstrings добавлены
[ ] Сложная логика объяснена
[ ] README обновлён если нужно
[ ] CHANGELOG обновлён
```

## Чеклист для ревью

### Архитектура
- [ ] Код следует архитектуре проекта
- [ ] Нет циклических зависимостей
- [ ] Модули слабосвязанные
- [ ] Ответственность разделена

### Производительность
- [ ] Нет N+1 запросов
- [ ] Кэширование где нужно
- [ ] Нет утечек памяти
- [ ] Асинхронность используется

### Обработка ошибок
- [ ] Конкретные исключения
- [ ] Логи с контекстом
- [ ] Graceful degradation
- [ ] Retry для внешних API

### Тестируемость
- [ ] Код тестируемый
- [ ] Нет скрытых зависимостей
- [ ] Моки используются
- [ ] Edge cases покрыты
